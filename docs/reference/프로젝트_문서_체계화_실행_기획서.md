# 프로젝트 문서 체계화 실행 기획서

> **목적**: 어떤 프로젝트든 첫날부터 체계적인 문서 관리로 시작하여, 장기 개발·협업·인수인계에서 혼란을 방지하는 프레임워크 제공  
> **적용 대상**: 개발 프로젝트, 연구 프로젝트, 팀 협업 프로젝트 등  
> **실행 시간**: 프로젝트 시작 후 1-2일 내 기초 구축, 이후 지속 유지

---

## 📋 1. 문서 체계 4단계 구조

### Foundation (기초/철학)
**목적**: 프로젝트가 흔들리지 않는 기준점 제공
- `Project_Memory.md`: 왜 이 프로젝트를 하는가? 최종 목표와 성공 기준
- `Architecture.md`: 어떻게 구현할 것인가? 시스템 구성과 데이터 흐름
- **특징**: 자주 변하지 않음, 모든 의사결정의 기준점

### Status (현황/진행)  
**목적**: 팀원 누구나 "지금 상황"을 즉시 파악
- `Current_Work.md`: 오늘 하고 있는 일, 막힌 부분, 다음 할 일
- `progress_flow.md`: 지금까지의 흐름과 앞으로 계획
- **특징**: 자주 업데이트, 실무진 소통의 허브

### Plans (계획/기획)
**목적**: 구체적 실행 로드맵과 우선순위 관리  
- `restructure_plan.md`: 기술적 리팩토링/개선 계획
- `YYYYQX_feature_plan.md`: 분기별/기능별 세부 기획
- **특징**: 중간 빈도 업데이트, 스프린트 단위 관리

### Reference (참고/가이드)
**목적**: 실행 시 참고할 매뉴얼과 예시
- `README.md`: 프로젝트 개요, 설치, 실행 방법
- `PRACTICAL_GUIDE.md`: 핵심 기능 사용법과 예시
- **특징**: 안정적, 새 팀원 온보딩용

## 🛠️ 2. 자동화 도구 구축 (3단계)

### 2.1 문서 점검 스크립트 (`tools/check_docs.py`)
```python
# 필수 구현 기능:
# 1. 필수 문서 존재 확인
# 2. 문서 수정일 추적  
# 3. 워크플로우 안내 (foundation → status → plans)
# 4. 주요 문서 자동 열람
# 5. CI 친화적 종료 코드
```

**핵심 로직**:
- 필수 문서 누락 시 경고 → exit code 1 (CI 실패)
- "철학 → 아키텍처 → 현황 → 계획" 순서 안내
- 마지막 단계 문서(plans 최신) 자동 열람

### 2.2 CI 자동 검증 (GitHub Actions)
```yaml
# .github/workflows/docs-check.yml
# - 매일 정기 실행 (한국시간 오전 9시)
# - PR/docs 수정 시 자동 트리거  
# - 필수 문서 누락 시 빌드 실패 알림
```

### 2.3 로컬 빠른 실행 스크립트
- **PowerShell**: `scripts/check_docs_ci.ps1` (상세 출력, CI 모드)
- **배치 파일**: `check_docs.bat` (간편 실행)

## 🚀 3. 프로젝트 시작 시 실행 순서

### 3-1. 첫날: 기초 문서 작성 (2-4시간)
1. **디렉터리 구조 생성**:
   ```
   docs/
   ├── foundation/
   ├── status/  
   ├── plans/
   └── README.md
   tools/
   scripts/
   .github/workflows/
   ```

2. **Foundation 문서 작성** (우선순위 1):
   - `Project_Memory.md`: 프로젝트 목적, 성공 기준, 왜 중요한지
   - `Architecture.md`: 전체 구조, 주요 컴포넌트, 데이터 흐름

3. **Status 문서 초안**:
   - `Current_Work.md`: "프로젝트 시작" 상태로 세팅
   - `progress_flow.md`: 향후 마일스톤 개요

### 3-2. 둘째날: 자동화 도구 설치
1. **`tools/check_docs.py` 구현**:
   - 기본 문서 목록 정의
   - 워크플로우 안내 추가
   - CI 모드 옵션

2. **CI 파이프라인 연결**:
   - GitHub Actions 워크플로우 추가
   - 로컬 실행 스크립트(배치/PowerShell)

3. **첫 검증**: `python tools/check_docs.py` 실행해서 정상 동작 확인

### 3-3. 지속 운영: 문서 최신화 규칙
- **매일**: `Current_Work.md` 업데이트 (진행 상황, 이슈)
- **매주**: `progress_flow.md` 검토 (마일스톤 달성도)  
- **분기별**: `plans/` 하위 계획 문서 갱신
- **중요 변경 시**: `Architecture.md` 수정

- 성공 기준:

- 주 사용자:
- 사용 시나리오:

- 우선순위 1:
- 우선순위 2:

```markdown

> **현재 스프린트**: [기간/목표]

## 🎯 이번 주 목표
- [ ] 목표 1
- [ ] 목표 2

## ⚙️ 진행 중
- **작업명**: 상태, 예상 완료일, 블로커

- **이슈명**: 현재 상태, 대응 방안

## 📅 다음 주 계획
- 우선순위별 할 일 목록
```

## 🔄 5. 다른 프로젝트 적용 가이드

### 5-1. 개발 프로젝트
- **Foundation**: 기술 스택, API 설계, 데이터베이스 구조
- **Status**: 개발 진행률, 버그 상황, 릴리스 준비도
- **Plans**: 스프린트 계획, 리팩토링 로드맵

### 5-2. 연구 프로젝트  
- **Foundation**: 연구 가설, 방법론, 예상 결과
- **Status**: 실험 진행, 데이터 수집 상황
- **Plans**: 실험 일정, 논문 작성 계획

### 5-3. 비즈니스 프로젝트
- **Foundation**: 사업 목표, 타겟 시장, 핵심 지표
- **Status**: 매출, 사용자 수, 주요 이슈  
- **Plans**: 마케팅 전략, 제품 로드맵
